Feedback & Programme Directory Page

Technical Description

The Feedback & Programme Directory page is divided into three key sections: a scrollable programme directory, a ratings and reviews section, and a JavaScript-enhanced feedback form. The page uses the shared HTML template structure with a <header>, <main>, and <footer>. Each section is wrapped in a semantic <section> element with an aria-labelledby attribute pointing to its heading, ensuring screen readers can identify each region clearly.

The scrollable programme directory uses CSS overflow-y: auto with a fixed max-height: 480px to create a scrollable container. Inside it, a CSS Grid with grid-template-columns: repeat(auto-fill, minmax(240px, 1fr)) arranges the four programme cards responsively without needing media queries. Each programme card is an <article> element containing a thumbnail image, a heading, description, and a button. Card hover effects use CSS transition on box-shadow and transform: translateY to lift the card on hover.

The feedback form uses <fieldset> and <legend> to group fields into two logical groups — Personal Details and Feedback Details. Input types used include type="text", type="email", type="tel", type="date", type="datetime-local", a <textarea> with maxlength="500", a <select> dropdown, and a <datalist> for searchable country name suggestions. A CSS Grid with grid-template-columns: 1fr 1fr creates a two-column layout for paired fields on wider screens.

JavaScript Implementation

JavaScript powers three interactive features. First, a live character counter listens to the input event on the message textarea and updates the remaining character count dynamically, changing colour to orange near 50 characters and red near 10. Second, a form validation function submitForm() checks all required fields on button click — using a regular expression to validate the email format, verifying dropdowns are not empty, and ensuring the message is at least 20 characters long. Error messages are shown next to invalid fields using DOM class manipulation. Third, real-time error clearing attaches input and change listeners to each field so errors disappear as the user corrects them. On successful validation, a confirmation message is revealed using classList.add('show').

Accessibility

Every image includes a descriptive alt attribute. All form fields have associated <label> elements linked via matching for and id attributes, ensuring screen readers announce the correct label when a field is focused. The <fieldset> and <legend> groupings provide additional context for assistive technologies. The confirmation message uses role="alert" so screen readers announce it automatically when revealed. The scrollable region carries an aria-label to identify it. Colour contrast between text and backgrounds meets WCAG 2.1 AA standards throughout.

Team Page

Technical Description

The Team Page presents all four group members using an interactive thumbnail grid. The layout uses CSS Grid with grid-template-columns: repeat(4, 1fr) on desktop, collapsing to two columns on tablet and one column on mobile via media queries. Each team member is a <div class="member-block"> containing a thumbnail container, a hover-reveal overlay panel, and a name bar.

The thumbnail image uses object-fit: cover to fill its container uniformly. The hover-reveal overlay expands below the thumbnail using a CSS transition on max-height — animating from 0 to 400px smoothly on hover. The overlay uses a dark background so text remains readable. The name bar below the thumbnail is always visible with a dark background and light text. On hover, the card border colour and box-shadow both transition to provide clear visual feedback.

A numbered student badge is positioned using position: absolute at the top-left of each thumbnail, styled as a circle using border-radius: 50%.

JavaScript Implementation

JavaScript handles keyboard accessibility by listening for keydown events on each member block. When the Enter or Space key is pressed, a CSS class expanded is toggled, replicating the same visual state as the CSS hover effect. This ensures keyboard-only users receive the same interactive experience as mouse users. The mobile navigation toggle is also controlled by JavaScript, adding and removing an open class on the navigation link list.

Accessibility

Each team member block has tabindex="0" so it can receive keyboard focus. A role="button" attribute communicates its interactive nature to screen readers. Each block carries a descriptive aria-label identifying the team member. All images include meaningful alt text. Decorative badge elements use aria-hidden="true". A visible focus style matching the hover effect ensures keyboard users can see which element is active at all times.

Content Page

Technical Description

The Content Page follows the shared HTML template with a <header>, <main>, and <footer>. The main area uses a two-column CSS Grid — grid-template-columns: 220px 1fr — placing a sticky internal navigation sidebar on the left and the article content on the right. The sidebar uses position: sticky with top: 84px so it stays visible while the user scrolls through the long page.

The content is divided into seven clearly labelled <section> elements, each with a unique id that serves as an anchor target for internal navigation links. One <h1> provides the page title, followed by <h2> and <h3> sub-headings in correct hierarchical order. At least four relevant images are included, each styled with object-fit: contain, width: 100%, height: auto, border-radius, and box-shadow. A two-column image grid uses grid-template-columns: 1fr 1fr for side-by-side image layouts. The "Go to Top" button is a plain HTML anchor <a href="#top"> with position: fixed — no JavaScript required.

Accessibility

The internal navigation sidebar uses an <aside> element with aria-label="Page sections navigation". All images carry descriptive alt text. The active navigation link is marked with aria-current="page". Headings follow a strict logical hierarchy. The page uses high-contrast colours throughout, and the layout collapses to a single column on mobile devices via a media query.

Challenges and Lessons Learned

Challenge 1 — JavaScript Form Validation

Building a form validator that displayed error messages next to specific fields — rather than a generic browser alert — was more complex than expected. Managing multiple error states simultaneously while also clearing them in real time required careful planning of DOM selection and conditional logic.

Resolution: I created a reusable helper function setError(fieldId, errorId, show) that adds and removes error classes and messages. I then attached separate input and change event listeners to each field to clear errors as the user corrected them, providing a smooth and responsive experience.

Challenge 2 — Team Page Hover Overlay Clipping

The initial approach placed the overlay inside the thumbnail using position: absolute. However, the overflow: hidden property needed on the thumbnail to clip the image was also clipping the overlay, preventing it from fully expanding and showing all content.

Resolution: I restructured the overlay to sit below the thumbnail as a sibling element. Using max-height: 0 transitioning to max-height: 400px on hover created a smooth reveal effect without any clipping, and the layout became cleaner overall.

Challenge 3 — Scrollable Section Height

Setting the right fixed max-height for the scrollable programme section was difficult — too small cut off content, too large defeated the purpose of scrolling.

Resolution: After testing multiple values, I settled on 480px — tall enough to show around two rows of cards and clearly suggest more content is available, without dominating the page layout.

Lessons Learned

These challenges taught me to plan the DOM structure carefully before writing CSS — particularly when properties like overflow: hidden affect child elements unexpectedly. I also learned that reusable JavaScript functions significantly reduce code repetition and make logic easier to maintain and debug.

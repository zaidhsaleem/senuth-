Content Page — Validation Report

The Digital Inequality & Access Content Page was validated using the W3C Markup Validation Service and passed with no errors. The validation confirmed that the page uses a correct HTML5 document structure with a single <h2> as the article heading, followed by properly ordered <h3> sub-headings across all four sections. All images carry meaningful alt attributes describing their visual content in the context of digital inequality. Internal anchor links in the in-page <nav> element were confirmed to correctly target their corresponding id attributes on each section. The embedded <style> block uses valid CSS properties throughout, including linear-gradient, display: flex, and backdrop-filter. The Go To Top button is implemented as a valid HTML button element controlled by a short JavaScript scroll handler.

Action Impact Simulator Page — Validation Report

The Action Impact Simulator page was validated using the W3C Markup Validation Service and passed with no errors. Reflection on the Validation Process

Running all three of my pages through the W3C Markup Validation Service was one of the most useful steps in the development process. It highlighted issues that were completely invisible in the browser but would have affected accessibility, search engine indexing, and cross-browser consistency — reinforcing that a visually correct page is not necessarily a structurally correct one.

One issue caught during validation on the Content Page was a missing closing tag on one of the <section> elements, which the browser had silently corrected but the validator correctly flagged. Fixing this ensured the document structure was clean and the anchor links in the in-page navigation targeted the right sections reliably across all browsers.

On the Action Impact Simulator, an early draft used <div> elements for the action cards rather than <label> elements. The validator highlighted that the checkboxes lacked proper label associations, which would have made them inaccessible to screen reader users. Restructuring each card as a <label> wrapper resolved both the validation issue and the accessibility concern simultaneously.

Overall, the validation process confirmed that all three pages comply with HTML5 standards and are free from errors. It was a valuable reminder that writing semantic, standards-compliant markup from the start saves significant debugging time later in development.

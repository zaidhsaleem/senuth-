Content Page — Digital Inequality & Access

Technical Description

The Content Page (content_ST2.html) is built using semantic HTML5 elements to produce a well-structured, long-form article. The page uses an <article> tag as the primary content wrapper, divided into four <section> elements — each assigned a unique id — covering: The Digital Divide, Infrastructure & Connectivity, Affordability Barriers, and Digital Literacy & Education. Each section id acts as an anchor target for the in-page navigation.

An in-page <nav> element at the top of the article contains anchor links using href="#section-id", allowing users to jump directly to any section without scrolling manually. This improves usability significantly on a long-form page. The CSS styling relies entirely on shared style.css custom properties such as var(--primary-color), var(--surface-color), and var(--radius-lg), ensuring visual consistency across the whole website without duplicating global rules.

Two additional components were defined in a page-specific embedded <style> block: a stat box, which uses display: flex and a linear-gradient background to present key statistics prominently; and a highlight block, which uses a left border and italic styling to draw attention to important quotes and mission statements. Side-by-side images use calc(50% - 1rem) widths inside a flex-wrap: wrap container, so they stack automatically on smaller screens without any additional media queries.

Accessibility

All images on the page carry descriptive alt attributes that convey context to screen reader users. The in-page navigation uses a semantic <nav> element, making it identifiable as a navigation landmark by assistive technologies. Heading levels follow a strict logical hierarchy from <h2> down to <h3>, aiding screen reader users in understanding the page structure. Colour contrast between all text and background combinations meets WCAG 2.1 AA standards. The fixed Go To Top button provides a clear, keyboard-accessible method for users to return to the top of the page without excessive scrolling.

Action Impact Simulator Page

Technical Description

The Action Impact Simulator page (ais.html) is structured around a single <div class="simulator-container"> placed inside the shared <main class="container"> layout. Six action cards are built as <label> elements each wrapping an <input type="checkbox">, a small image, an action-details div, and a visible point badge — creating a fully clickable card that toggles its checkbox without requiring a separate click target.

Each checkbox carries a data-score attribute holding its point value. The JavaScript reads these values using getAttribute('data-score') and accumulates a total on every change event. The score is displayed inside a .score-board div styled with a CSS linear-gradient background. Three impact levels are defined by score thresholds — each triggering a different message and a dynamic background colour change applied directly to document.body.style.background via JavaScript, satisfying the requirement that background changes must be handled through JavaScript and not CSS alone.

A .selected CSS class is toggled onto checked cards using classList.add and classList.remove inside the event handler, providing clear visual feedback to the user when an action is selected. The page-specific styles — including .action-img, .action-points, .score-board, and .selected — are all defined inside an embedded <style> block on this page rather than in the global stylesheet.

JavaScript Implementation

The core logic runs inside a DOMContentLoaded event listener. A calculateScore() function iterates over all checkboxes using querySelectorAll('.impact-cb'), sums the scores of checked items, updates the score display with a brief CSS scale animation, and then evaluates the total against three thresholds to assign the correct impact level message and page background. Each checkbox has a change event listener attached that calls calculateScore(), so the UI updates in real time with every selection or deselection.

Accessibility

Using <label> elements as card wrappers ensures that clicking anywhere on a card — including the image or text — activates the associated checkbox, improving usability for all users. Each image carries a descriptive alt attribute. The score board uses clear, high-contrast colours at all three impact levels. The page inherits the shared navigation structure with keyboard-accessible links, and all interactive elements are reachable via the Tab key.

Splash Screen Page

Technical Description

The Splash Screen (splash.html) uses a single full-viewport <div class="splash-screen"> that covers the screen using position: fixed, width: 100%, and height: 100%. It is layered using three stacked child elements: a .splash-background div carrying the CSS gradient, a .splash-overlay div providing a semi-transparent dark overlay for text readability, and a .splash-content div holding all visible content including the title, loader, mission statement, countdown, group members list, and skip button.

The animated loader uses three <div class="loader-dot"> elements styled as circles with staggered animation-delay values on a loaderBounce keyframe animation, creating a bouncing dots effect without any JavaScript. The countdown display uses a <span id="countdown"> updated each second by a JavaScript setInterval timer. A <meta http-equiv="refresh"> tag provides a fallback redirect in case JavaScript is disabled. The skip button calls a skipIntro() function that clears the interval and immediately triggers the redirect.

All entrance animations — title slide-in, content fade-in, and staggered element reveals — are handled purely through CSS @keyframes and animation-delay, keeping the JavaScript focused solely on the countdown and redirect logic.

Accessibility

The skip button provides users — particularly those using assistive technologies or who prefer reduced motion — with immediate control to bypass the timed intro. The group members list uses a semantic <ul> and <li> structure. Text contrast against the dark overlay meets WCAG 2.1 AA requirements. The page is fully responsive, with font sizes and padding scaling down at 768px and 480px breakpoints via media queries.

Challenges and Lessons Learned

Challenge 1 — Responsive Side-by-Side Images

On the Content Page, the two side-by-side images overflowed their container on narrow screens when given fixed widths. The layout broke below around 600px.

Resolution: I wrapped both images in a display: flex container with flex-wrap: wrap and set each image width to calc(50% - 1rem). This allowed them to sit side by side on wider screens and automatically stack to full width on smaller viewports, removing the need for explicit media queries.

Challenge 2 — Action Simulator Background Change via JavaScript

Initially the background colour changes on the simulator were handled using CSS class toggling, which the requirements explicitly stated must be done through JavaScript instead.

Resolution: I moved the background logic entirely into the calculateScore() JavaScript function, applying gradient values directly to document.body.style.background based on the score threshold reached. This satisfied the requirement and also made the three impact levels much more visually distinct from each other.

Challenge 3 — Splash Screen Countdown and Meta Refresh Conflict

The <meta http-equiv="refresh"> tag and the JavaScript setInterval countdown occasionally caused a double-redirect on slower connections, where the meta refresh fired slightly before the JavaScript redirect completed its fade-out transition.

Resolution: I set the meta refresh duration to match the countdown value exactly, and added a 500ms delay inside the JavaScript redirect to allow the CSS fade-out transition to complete visually before the page change. This ensured a smooth experience in all tested browsers.

Lessons Learned

These challenges reinforced the importance of reading requirements carefully before deciding on an implementation approach — particularly around which features must be JavaScript-driven versus CSS-driven. I also learned that testing across different screen widths early in development catches layout issues before they become embedded in the structure and harder to fix.
